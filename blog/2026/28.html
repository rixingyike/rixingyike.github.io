Vide Coding 经验总结，核心五点

以下内容都是我进行 Vide Coding 后的亲自经验总结，欢迎批判或补充。相信真正使用过 Vide Coding 的人都会会心一笑或拍掌叫好。真正做到这六点，编码效率将像开火箭一样又快又稳！

1，目录化。

不依赖于环境，不依赖于系统常量，目录化的代码、配置、资源、类库实现自足，可以实现从一个系统转到另一个系统完全可以在一键 setup 后就可以运行。项目一般有一些配置文件和脚本，例如 config.toml 配置、dev.sh 调试脚本、build.sh 编译脚本。

对于 Agent 而言，它无法像人类一样自己去配置复杂的环境变量或摸索依赖路径。自包含的目录和标准化的启动脚本（如 dev.sh、build.sh）相当于给了 Agent 一个明确的“抓手”和“入口”。当 Agent 迷失或遇到报错时，它可以直接调用这些标准脚本进行一键调试，减少它“盲人摸象”式的猜忌。

2，结构化。

结构化可以让提示语内容更富于逻辑。

与 Agent 沟通时，非结构化的大段文字极易触发“幻觉”或让它漏掉关键信息。将提示语（Prompt）和项目需求结构化（例如使用 Markdown 标题、JSON 格式、XML 标签来清晰划分【背景】、【目标】、【输入】、【约束】），能极大降低沟通成本。提示词越像伪代码或清晰的大纲，Agent 输出的代码结构就越严谨、精确度就越高。

3，模块化。

项目无论是哪种编程语言，都要遵守专一职责原则，一个函数只干一件事，一个文件只干一类事。有一类文件是路由编排文件，负责将其它文件组织在一起，例如 main，这类文件的作用就是路由编排，它们也是只负责一类事。

模块化是破解 Agent **“上下文窗口限制（Context Window）”和“注意力遗忘”**的终极武器。面对动辄几千行的大文件，Agent 极容易“顾此失彼”或“写到后面忘了前面”。将职责拆分到极细的模块中，每次让 Agent 只聚焦并修改一个几百行的小文件，既能节省 Token，又能保证它对当前代码逻辑的 100% 掌控。

如果我们不这样要求，Agent 最喜欢做的事就是将所有代码都写在一个文件中。

4，可测试、可验证。

所有已经完成的功能都要可验证，最好都有测试文件，可以一键启动测试。对于已经验证过的文件，最好有一个 lock 属性，当 lock 等于 true 的时候，文件便是只读的，只能使用，不能修改。一个好的系统，稳定的系统，支持扩展而不支持修改，如果旧功能达不到新阶段需要了，可以扩展出新功能。但新功能在验证后，也要 lock 住。lock 住的功能越多，系统越稳定。项目不怕大，就怕混乱；只要始终变化的只是当前关注的新功能点，项目添加多少功能都不怕。

这是用 Agent 编程最核心的护城河！Agent 有时会为了迎合新需求，擅自“自作聪明”地修改甚至重构之前已经跑通的老代码。

  • 测试是防线： 让 Agent 先写测试用例（TDD 开发模式），用一键测试脚本来充当验收裁判，而不是人眼费力去一行一行 Review 代码。
  • Lock 是底线： 严格遵循开闭原则（对扩展开放，对修改封闭）。告诉 Agent “哪些文件已被 Lock 绝对禁止修改，只能调用或继承”，能大大减少 Agent 改坏系统的可能性。

5，文档化。

当功能完成后，最好要分模块细化出汇报文档。文档是逻辑的结晶，有时候，只要有完备的文档，就可以再造系统；有时候，文档也是进一步完善系统的基础。

文档不仅仅是给人类看的，在 AI 时代，文档就是 Agent 的“长期记忆”。当项目庞大到无法一次性喂给 Agent 时，模块说明、接口文档、架构图说明就是最好的上下文。哪怕由于超时导致对话窗口重置，只要抛出这些浓缩的文档结晶，随时可以唤醒一个新的 Agent 实例立刻接手工作，实现“代码与记忆的无损传递”。

6，原子化。

小步快跑，积少成多。千万不要指望 Agent 一次对话就写出一个庞大的系统。将需求拆解成极小的“原子任务”，比如只写一个接口，或只画一个 UI 组件，一步一步完成，一步一步验证。结合第 4 点，每完成一个小步就立即验证、Lock 归档、提交 Git Commit。这样即使 Agent 某一步“发散跑偏”了，回滚的成本也极低。
